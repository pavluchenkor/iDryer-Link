# Обновление потока Claim/MQTT (для разработчика)

Дата: 2026-03-04

## Цель изменения

Сделать строгий порядок выхода устройства в рабочий cloud-режим:

1. Сначала `Hello` от RP2040.
2. Потом MQTT авторизация.
3. Потом `info`.
4. И только после этого весь остальной обмен.

Если RP2040 не подключен или молчит, в рабочий MQTT-режим не выходим.

## Обязательные правила

1. До `Hello` от RP2040:
- запрещен переход в рабочий MQTT-обмен;
- не публикуем `telemetry/status/weights/rfid/config`;
- не обрабатываем входящие MQTT-команды.

2. Исключение до `Hello`: разрешен только явный Claim по команде пользователя:
- `START_CLAIM` (USB Serial) или `ClaimStart` (UART),
- `POST /devices/provision` (если нет `token`),
- `POST /devices/register` (получение PIN),
- polling `GET /devices/check-claim/{token}` (получение `deviceId`),
- сохранение `token/deviceId` в NVS.

3. После успешного Claim, но без `Hello`:
- MQTT не поднимать;
- оставаться в состоянии ожидания MCU (`WaitHello`/эквивалент).

4. Когда есть `Hello` + `token` + `deviceId`:
1. `MQTT CONNECT`
2. `SUBSCRIBE idryer/{serial}/commands/#`
3. публикация `info` первым сообщением
4. только после успешного `info` разрешить:
   - публикацию остальных топиков;
   - обработку входящих MQTT-команд.

5. Команды, пришедшие до успешного `info`, игнорировать (без буферизации).

6. На каждом MQTT reconnect повторять тот же порядок:
`CONNECT -> SUBSCRIBE -> info -> остальной обмен`.

## Ожидаемый результат

- Без RP2040 устройство не \"ломится\" в рабочий realtime-режим на портал.
- После восстановления UART/MQTT порядок всегда детерминированный.
- Backend получает `info` как обязательную первую рабочую публикацию.
